SMACSS

Scalable and Modular Architecture for CSS

  • Increase semantic value of a section and html and content
  • Decrease the expectation of a specific html structure
  • Decrease coupling between HTML and CSS

Base

  • Apply to elements and element hierarchies
  • These will NOT include class or id selectors
  • Mostly used for default rules
  • Check out Reset stylesheets

Example

body { font-size: 10px; }

header { width:100%; }

Layout

  • Everything can broken into components. These components should be small and can easily be reusable.
  • A header, footer, or main style rules would be a layout but a title and logo in the header would be more of a module.

Before

div#featured ul { margin: 0; padding: 0; list-style-type: none; }
div#featured li { float: left; height: 100px; margin-left: 10px; }

After

.l-grid { margin: 0; padding: 0; list-style-type: none; }
.l-grid > li { display: inline-block; margin: 0 0 10px 10px; }

Module Rules

  • Smallest page components found on the site
  • The goal that modules are self contained and can be moved all over the site
  • Try not to use descendant selectors
    • Increase Coupling
    • Reduce Reuse
  • Can fit inside other modules and typically inside layouts rules

Before

.box { width: 200px; }
.box ul { width: 100px; }
#sidebar .box ul { width: 200px; }

After

.box { width: 200px; }
.box-contstrained { width: 100px; }

State rules generally apply different rules to the same element based on certain conditions

  • Adding/Removing a class through JavaScript
  • Using a pseudo-class which affect special elements such as descendants or siblings
  • Media Queries
Theme rules are similar to state rules but tend to contain colors and typography and a usually a client wide style used across sites
